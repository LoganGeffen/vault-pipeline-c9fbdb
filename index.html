A Structure — what talks to what

The parts of the pipeline and where each one runs: the member’s browser, the Supabase backend, and GitHub. The numbered arrows trace one job in order, 1 through 14; the list below is one line per arrow. Git holds each module’s definition, and the compute_modules table lists the modules currently enabled, generated from those definitions. Results are written back to the database only by job-callback, not by the runner. The green arrow is the separate release step that syncs merged modules into compute_modules. The grey stale sweep and the dashed grey EC2 box are background and future work, not part of a single run. The EC2 lane is not built yet.

  1. The browser loads the list of enabled modules and the input form from compute_modules.
  2. The member submits the chosen module and one line of text to submit-job.
  3. submit-job checks compute_modules to confirm the module is enabled.
  4. submit-job verifies the login and resolves the member’s org from identity (no org, no run).
  5. submit-job inserts a queued row in the jobs table.
  6. submit-job triggers run-module.yml with a repository_dispatch event.
  7. run-module.yml reads the manifest, writes the input to inputs.json, and runs run.py on Python 3.9. The exit code is the result: 0 means success.
  8. run-module.yml sends the exit code and the output file to job-callback.
  9. job-callback writes the output object to private Storage at <job_id>/<attempt>/<file>.
  10. job-callback sets the jobs row to succeeded or failed.
  11. The browser polls its own org’s rows in jobs to watch the status.
  12. The member requests the download from get-result.
  13. get-result confirms the job succeeded and that the member’s org owns it.
  14. get-result signs the exact stored object and returns a link that expires after 60 seconds.

B Flow — the order it happens

The same job as a detailed sequence, read from the top down. It shows the internal replies that figure A leaves out: the 202 ticket handshake, the running status update before run.py, and the row-level checks. The blue messages are the same forward path numbered in figure A.

C Data — the tables

The tables and storage the pipeline uses. compute_modules is generated from the git manifests and is not edited directly. The jobs table records each run and the org it belongs to. jobs.module points to compute_modules.slug, but that link is checked in code, not enforced as a database foreign key. Each successful run writes one output file to private Storage at <job_id>/<attempt>/<file>.
