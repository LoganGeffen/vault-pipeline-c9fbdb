How a science module runs now

One generic runner accepts a manifest-defined value or private file, resolves the module into either a direct Python lane or a sandboxed container lane, executes its wrapper, and returns one private result through an organization-scoped, time-limited download.

AStructure · current components

Git manifests are canonical; compute_modules is their enabled runtime projection. The module runner never writes to Supabase directly. job-callback is the machine-authenticated gateway for input retrieval, lifecycle updates, private result storage, and best-effort terminal input cleanup. Registry visibility, dispatchability, and entitlement are three separate things: ui_visible is presentation, enabled is dispatchability, and the tier map in module-entitlements.ts is authorization.

Figure A — C4-style component and system structure view: containers, edge functions, stores, and the two execution lanes.

Narrow automation boundary: a qualifying protected-main push runs release-modules.yml, which projects module manifests into compute_modules and nothing else. It does not deploy edge functions, apply SQL, or publish the frontend. Entitlement policy lives in edge-function source, so changing a module's required tier is a function redeploy, not a database write. Git, Supabase, and the Lovable-connected publication surface remain separate truth surfaces.
No stale-job sweeper exists. Source comments refer to a reaper as the backstop for a dispatch whose delivery is unknown, but nothing implements one. Such a job stays queued and keeps counting against the period quota.

BFlow · one file job end to end

This is the file-input superset path. Value modules skip private input staging and retrieval entirely — their value travels in the dispatch payload — then join the same execution, callback, result, and download path. Authentication, registry and input checks, organization membership, and entitlement all precede the reservation; the quota reservation and the jobs insert are a single atomic RPC, so a rejected submission leaves no ticket behind.

Figure B — UML sequence diagram: a representative file job, from form load to signed download.

Failure and quota semantics: dependency, retrieval, workflow, callback, or delivery failure is failed/infra; a non-zero run.py exit is failed/science. Each accepted paid submission reserves one run against the period. A run stops counting only when it is failed, staged infra, and never started. Queued jobs whose delivery is unknown, infra failures that already started, and every science failure all continue to count.
Input cleanup is best-effort. An accepted succeeded or failed terminal callback attempts cleanup after callback processing; a definite dispatch rejection also attempts cleanup. Lookup and delete failures are logged rather than retried, while delivery-unknown or never-arriving callbacks can leave the object in place. There is no bucket lifecycle policy behind it.

CData · canonical definition versus runtime state

jobs records execution state and doubles as the quota ledger; compute_modules records the enabled UI/runtime contract. They are deliberately separate, and neither carries tier. Private input objects are transient; private result objects are returned only through a membership-checked signed URL.

Figure C — Entity-relationship and private-storage state view: tables, buckets, quota semantics, and the Git-owned policy that sits outside the database.

Where a signed URL becomes a bearer capability: get-result re-verifies the caller's bearer token and organization membership, then issues a 60-second signed URL for the exact result object. From that point the URL alone grants the read — it is not bound to the session, user, or address, and there is no revocation path. get-result checks membership, not entitlement, so an organization that later loses a tier retains access to results it already produced.

DBuild and ship · the contributor boundary

A runnable module is three files, or four

modules/<slug>/
├── manifest.yaml      # one input + one output + runtime
├── requirements.txt  # public-PyPI pins; may be empty
├── run.py             # inputs.json → declared output
└── Dockerfile         # container modules only

run.py is executed as a process, never imported. It exits 0 on success and non-zero on a science failure. It does not update Supabase. manifest.yaml declares the runtime lane — python with a quoted major.minor version, or container with a module-local Dockerfile.

No shared-code edit for a standard module

The contributor owns only the module directory, local science smoke tests, manifest/release dry-runs, and a feature-branch PR.

Do not edit submit-job, run-module.yml, job-callback, Supabase rows, or the Runs frontend for a standard module.

Tier is the exception. A module's required tier is not manifest data — it lives in _shared/module-entitlements.ts and is owned separately from the module contract.

Two lanes, two isolation stories. Python modules run directly on the hosted runner — there is no container isolation and no network restriction on that lane. Standard container modules run with --network none, --cap-drop ALL, --security-opt no-new-privileges, a read-only root filesystem, and a 64 MB tmpfs. protox-3 is the sole container-egress exception: an internal broker restricted to tox.charite.de:443, with a 512 MB tmpfs. Because NetInfer is a Python module reaching external services over ordinary runner egress, ProTox is not the system's only egress consumer — only its only brokered container.
Pinning varies; do not read it as uniform. The Python runtime is manifest-declared to major/minor only, so the patch level floats with the runner image. Dependencies are exactly pinned where declared, but several modules declare none. Container builds are module-local with no shared base, and only some are digest-pinned.
Recovery only: an authorized operator may rerun the release workflow or use the apply script from a clean synchronized main. Neither is the normal contributor release path.
