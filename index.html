How a science module runs

One generic runner takes a manifest-declared value or private file, executes the module in a pinned Python or container runtime, and returns exactly one small result through an organization-scoped, short-lived signed download.

AThree separate truths

The single most common misreading of this system is that merging to main ships something. It does not. Git, Supabase, and Lovable are three independent surfaces, and a green merge proves only the first one.

Git — tracked source

origin/main is the truth of what the source says. Manifests, runner workflows, and the entitlement policy file all live here.

A manifest that reads enabled: true proves tracked intent — not a registry release, not a successful run, not scientific accuracy, and not publication.

Supabase — deployed behaviour

Holds the deployed edge functions, the database, private Storage, and the live module registry. A source merge does not deploy any of it.

The live function enforces whatever was last deployed. Changing the policy file in Git has no effect until that function is redeployed.

Lovable — editor and publication

Holds frontend editor and published-site state. Pushed commits sync into the editor; they do not publish production.

Publication is an explicit, separately authorized action. A merged frontend change does not prove the live site was published.

The one narrow exception. A push to protected main touching module paths runs release-modules.yml, which validates every canonical manifest and then projects them into the live compute_modules registry. That is the only automatic Git-to-live projection in the platform, and it writes exactly one table. It deploys no SQL, no edge functions, and no frontend.

How the registry projection behaves

The release job checks out canonical main, asserts it really is on main, validates the manifests, then posts a single sync_modules payload. Rows are upserted by slug; any module absent from the payload is disabled, not deleted. The workflow then compares the returned synced count against the expected count and fails the run on mismatch.

Tier access is deliberately not a manifest field and is deliberately not in the registry, so a registry release never depends on billing state.

BWhat happens when a member submits

A signed-in member picks a module in the app and submits one input. Five hops later they download one file. Every trust decision happens server-side.

1 · BrowserMember submits one value (JSON) or one file (multipart) for a chosen module.
2 · submit-jobVerifies the login token, resolves the org, checks entitlement and quota, mints the job ticket, stages any file privately.
3 · run-moduleDispatched workflow resolves the declared runtime, materializes the input, and executes the module wrapper.
4 · job-callbackRunner reports status with a shared secret; the single result is stored in private Storage and the staged input is cleaned up.
5 · get-resultMember re-authenticates, membership is rechecked, and a 60-second signed URL is returned.
The ticket exists before anything else can go wrong. submit-job creates the job row before staging files or dispatching, so there is never a GitHub run without a matching row. If the dispatch is definitively rejected the ticket is failed immediately; if delivery is merely uncertain — a 5xx or a timeout — the row is deliberately left queued for the runner callback or the reaper, because the system must never report "failed" for a job that may in fact be running.

DThe generic module contract

Every module — trivial echo or containerized external integration — obeys the same narrow v1 contract. That narrowness is what lets one runner serve all eleven.

Exactly one input

Either one string value, or one required file up to 5 MB. Multiple inputs, optional files, and multiple files are all unsupported in v1.

File bytes go to submit-job, which stages them in a private bucket. The runner downloads them to a fixed local path and writes something like {"image":"input/image"} into inputs.json.

The module opens a local path. It never receives a browser filename, a Storage credential, or an arbitrary URL.

Exactly one output

One file under output/, media type text/csv or text/plain, no larger than 5 MB. The runner rejects symlinks and path escapes before delivery.

The result travels back base64-encoded in the callback, which is why the size cap matters: this transport suits small results only. Large real outputs would need a signed upload path instead.

Dependency boundary

Python modules may use the standard library, code committed with the module, and exact package==version pins from their own requirements.txt. Ranges, URLs, editable installs, and environment markers are outside the contract.

Modules must not rely on whatever happens to be preinstalled on the runner. Container modules may additionally install their environment in a secretless Dockerfile build.

Sandbox

Containers run as the host user with --network none, all Linux capabilities dropped, no-new-privileges, a 256-process limit, and a read-only root filesystem. /tmp is a bounded 64 MB noexec,nosuid tmpfs.

/job/inputs.json and /job/input are mounted read-only; /job/output is the only writable persistent mount.

A module cannot request network access or extra permissions through its manifest. That would require a separately reviewed platform exception.

A declared MIME type is not proof of content. The browser and backend enforce the declared media type and byte limit, but the module wrapper is still required to decode the file and validate its real format itself.

EWhat the server enforces

The browser is never trusted for identity, organization, tier, or price. Three independent gates run inside submit-job, in order, before a job row exists.

1 · Membership and organization

The caller's login token is verified server-side; anonymous or invalid callers are rejected before any insert. The organization is then resolved from the member's own profile and memberships — never from a value the browser supplied. A user with no organization is refused, and a user whose several memberships leave the current organization ambiguous is asked to choose rather than being silently assigned one.

2 · Tier entitlement

Module access lives in one Git-owned policy file, not in the manifest and not in the registry. Higher tiers include every lower tier. An entitling subscription must be trialing or active; anything else is refused with a distinct "requires subscription" versus "requires upgrade" answer.

3 · Billing-period submission quota

Before the ticket is minted, a single atomic database reservation both counts the submission against the organization's current billing period and returns the job id. Exhausting the quota returns a 429 carrying the used count, the limit, and the reset time. If billing state shifted underneath the request, it returns a 409 asking the caller to refresh rather than guessing.

Getting the result back

Results live in a private bucket. get-result is the only read path and is fail-closed: it re-verifies the login token, re-checks that the caller belongs to the job's organization, and returns a 60-second signed URL only for a job that actually succeeded. A job that is not yours and a job that does not exist return the identical response, so the endpoint cannot be used to discover other organizations' jobs.

FProTox-3 — the one egress exception

Every other container runs fully offline. ProTox-3 is the single platform-owned exception, and it is worth understanding precisely because it shows where the architecture's boundaries actually sit.

How the exception is built

ProTox-3 does not get general network access. It runs behind an isolated broker that permits exactly one destination: tox.charite.de:443. Everything else is refused.

That permission is expressed in the platform's own workflow branch — deliberately not in the module manifest, because manifests cannot declare network policy.

The module drives a headless browser against the provider's page, parses the resulting tables, and writes one CSV. Browser assets used by that page are checksum-pinned in the image.

What it is not

It is an integration with an external public service — not an SSC-owned toxicity model, and not an experimental safety test.

Predictions are computational estimates, not measurements, diagnoses, or proof of safety. Their values and model coverage can change independently of SSC source.

The two-column input contract cannot carry dose, route, exposure duration, formulation, species, or measured context.

Inputs leave SSC. Submitted structures are disclosed to a third-party public service. The provider describes itself as free for academic and non-commercial use and places limits on automated queries; SSC's intended use, authorization, rate policy, and disclosure posture for submitted structures remain unresolved policy questions. Re-enabling the module did not resolve them. Use only authorized, non-confidential test compounds while those questions stay open.

Known fragilities, stated plainly

  • DOM coupling. The integration drives the provider's page structure, so a provider redesign can break it even while the public site still works for people.
  • Completeness is not guaranteed. The provider currently documents more prediction endpoints than the last authorized run returned. The runner preserves the rows it finds; row count is not a stable completeness guarantee.
  • Partial batches succeed. If at least one compound succeeds the job returns success, preserving the failure disclosure. Consumers must inspect the per-row status and error code rather than treating job success as full-batch success.
  • No recurring live check. CI builds and contract-tests the container but does not perform a recurring live-provider prediction, so provider drift can appear between authorized probes.
  • Validation is structural. Input checking covers CSV shape and bounds; it does not locally establish that a SMILES string is chemically valid before disclosing it to the provider.

GWhat a passing run actually proves

This is the distinction the whole architecture is organized around, and it is the easiest one to lose in a status update.

An execution receipt proves

That the pipeline ran end to end at that moment: the input was accepted and staged, the declared runtime was prepared, the wrapper executed, the output matched the declared contract, the callback stored it, the signed download worked, and staged input was cleaned up.

It does not prove

Scientific accuracy. A module can produce a perfectly contract-shaped CSV of wrong numbers. Nothing in this pipeline evaluates whether a prediction is correct.

Nor does it prove provider permission, provider stability, output completeness, or that the behaviour is currently deployed and published.

Execution proof is not scientific validation. Every claim this platform can make from a green run is a claim about plumbing. Scientific validity of any module's output is a separate question that this architecture deliberately does not attempt to answer.

Reading any status claim safely

  • A manifest saying enabled proves tracked intent, not a registry release.
  • A registry release proves the live registry row, not deployed functions or published frontend.
  • A merged PR proves source, not deployment and not publication.
  • A successful job proves execution, not accuracy.
  • Prices, tier placement, and submission limits are test placeholders pending approval.
