How a science module runs now

One generic runner resolves a module's declared runtime, executes it either as declared Python on the hosted runner or inside a sandboxed container, and returns one private result through a membership-checked download.

AStructure · current components

Git manifests are canonical; compute_modules is their enabled registry projection, currently 11 modules. Each manifest declares its own runtime: 6 run as Python directly on the hosted runner (3.9 for echo, reverse, image_summary, sequence_summary; 3.11 for fast-solv, netinfer) and 5 run as containers. The module runner never writes to Supabase directly; job-callback is the machine-authenticated gateway for input retrieval, lifecycle updates, private result storage, and terminal input cleanup.

Narrow automation boundary: a qualifying protected-main push runs release-modules.yml, which validates every manifest and projects the registry exactly — nothing more. It does not deploy SQL, edge functions, or the frontend; those still require their own explicit delivery paths.

BFlow · one file job end to end

This is the file-input superset path. submit-job enforces one fixed order: authenticate, validate the module contract, resolve membership, check tier entitlement, atomically reserve quota while creating the job row, optionally stage the file, then dispatch. Value modules skip private input staging and retrieval, then join the same execution, callback, result, and download path. Only file inputs ever enter private Storage; browser MIME metadata is not trusted as proof of file format.

Terminal semantics: dependency, retrieval, workflow, callback, or delivery failure is failed/infra; a non-zero wrapper exit is failed/science. Staged input deletion on a terminal callback is best effort — a failed cleanup is logged, not fatal. Quota is counted live from job rows rather than refunded: only failed/infra rows with no started_at are excluded from the period count.

CData · canonical definition versus runtime state

jobs records execution state; compute_modules records the enabled UI/input-output contract. They are deliberately separate, joined only by a validated logical slug. Organization membership and subscription tier gate submission; private input objects are transient, and private result objects are returned only through a membership-checked signed URL.

DBuild and ship · the contributor boundary

A runnable module is a manifest plus a wrapper

modules/<slug>/
├── manifest.yaml      # one input + one output + runtime
├── requirements.txt   # always required · exact pins; may be empty
├── run.py             # inputs.json → declared output
└── Dockerfile         # container runtime only

The wrapper is executed as a process, never imported. It exits 0 on success and non-zero on a science failure, and it does not update Supabase.

No shared-code edit for a standard module

The contributor owns only the module directory, local science smoke tests, manifest/release dry-runs, and a feature-branch PR.

Do not edit submit-job, run-module.yml, job-callback, Supabase rows, or the Runs frontend for a standard module. Tier access is not a manifest field — it lives in one server-side list.
