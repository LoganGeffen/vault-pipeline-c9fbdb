How a science module runs now

AStructure · current components

Git manifests are canonical; compute_modules is their enabled runtime projection. The module runner never writes to Supabase directly. job-callback is the machine-authenticated gateway for input retrieval, lifecycle updates, private result storage, and terminal input cleanup.

Narrow automation boundary: a qualifying protected-main push runs release-modules.yml and synchronizes module metadata. SQL, edge functions, and frontend publication still require their own explicit delivery paths.

BFlow · one file job end to end

This is the image_summary superset path. Value modules skip private input staging and retrieval, then join the same dependency, execution, callback, result, and download path. The ticket is created before dispatch; browser MIME metadata is not trusted as proof of file format.

Failure semantics: dependency, retrieval, workflow, callback, or delivery failure is failed/infra. A non-zero run.py exit is failed/science. Terminal callbacks remove staged file inputs.

CData · canonical definition versus runtime state

jobs records execution state; compute_modules records the enabled UI/runtime contract. They are deliberately separate. Private input objects are transient; private result objects are returned only through a member-checked signed URL.

DBuild and ship · the contributor boundary

A runnable v1 module is three files

modules/<slug>/
├── manifest.yaml      # one input + one output
├── requirements.txt  # exact public-PyPI pins; may be empty
└── run.py             # inputs.json → declared output

run.py is executed as a process, never imported. It exits 0 on success and non-zero on a science failure. It does not update Supabase.

No shared-code edit for a standard module

The contributor owns only the three module files, local science smoke tests, manifest/release dry-runs, and a feature-branch PR.

Do not edit submit-job, run-module.yml, job-callback, Supabase rows, or the Runs frontend for a standard v1 module.
